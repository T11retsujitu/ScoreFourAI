# 立体4目並べ AI プロジェクト 設計メモ

> 重力あり版（Score Four）を対象に、**最小労力で最強の解析エンジン**を作るための設計方針。
> 最優先ゴール：**α-β 探索中心の解析エンジン**。

---

## 0. ゴールと成果物

| 項目 | 内容 |
|------|------|
| 最優先ゴール | 最小労力で最強の**解析エンジン**（→ α-β 中心） |
| 最終的に欲しいもの① | **定石** |
| 最終的に欲しいもの② | **対戦・解析用 AI** |

**重要な順序の訂正**：要求リストは「①定石 → ②AI」だが、実装上の依存は**逆**。
定石は手で書くものではなく、**強いエンジンの出力として抽出される派生物**。
したがって作る順は **②エンジン → ①定石**。

---

## 1. 対象ゲームの定義

対象は **Score Four（重力あり立体四目並べ）**であり、自由配置版の **Qubic とは別物**。

- **重力ルール**：16 本の柱のうち満杯（4 個）でないものを選び、コマは最下段の空きへ落ちる。高さは選べない。
- **勝利条件**：縦・横・斜めいずれかの直線に同色 4 つ。盤面全体の勝利ラインは **76 本**。

| 勝利ライン種別 | 本数 |
|----------------|------|
| 軸に平行（縦・横・奥行き） | 48 |
| 平面内の斜め | 24 |
| 立体対角線 | 4 |
| **合計** | **76** |

### 探索アルゴリズムにとっての性質（方式選択の根拠）

- **分岐数 ≤ 16**（空き柱の数のみ、一方向に減少）
- **最大手数 ≤ 64**
- 勝敗は**戦術的な詰み**（ダブルリーチ・ツークツワンク）
- 盤面は **D4 対称（8 重の対称性）**
- ゲームとしては**未解決**（自由配置版 Qubic は 1980 年に先手必勝が証明済み）

> → 「探索が破綻しない側」の規模。確率的近似（AlphaZero）より、決定的に深く読める α-β の方が実質的に強く、軽い。

## 2. 参考事例：qweral 氏の取り組み

note の qweral 氏は、立体四目並べを段階的に攻略している。

1. **戦略の整理** → 最終的に AI の評価関数設計へ接続
2. **機械学習で解く** → 盤面から勝率を予測するモデル
3. **最強 AI の作成** → 「世の中で一番強い」と称するエンジン
4. **AlphaZero の応用** → コードは主に AI（Antigravity）に生成させている

流れは **手作り評価関数＋探索 → ML 勝率予測 → AlphaZero**。
王道（探索）から重い手法へ拡張した構成であり、AlphaZero は「最強の更新」と研究的興味によるもの。**最初の一手としては過剰**。

---

## 3. 方式選択：なぜ α-β 中心か

- このゲームは分岐 ≤16・深さ ≤64 と探索向き。AlphaZero が本領を発揮するのは囲碁（分岐約 361・状態 10^170）のように探索が破綻する規模で、立体四目はその真逆。
- α-β は**決定的・デバッグ可能・CPU のみ**で動き、戦術的な詰みの読み切りに強い。
- AlphaZero は GPU・自己対戦パイプライン・学習コストが必要で、本ゲームでは費用対効果が悪い。**頭打ちになった後 or 学習目的のときだけ**着手すればよい。

---

## 4. エンジン構成（費用対効果の高い順）

1. **ビットボード ＋ 76 ラインのマスク**
   各色を 64bit 整数で保持。勝利判定は事前計算した 76 本のマスクで `(player & line) == line`。
   さらに「直前に置いたセルを通る線だけ」を見れば**増分判定**で最速。
2. **置換表（Zobrist hash）** ← 効果が最大
   深さ・評価値・フラグ（exact / lower / upper）・最善手を保存。
3. **脅威ベースの強制手枝刈り** ← 実戦的な強さを最も伸ばす
   相手に「次に 4 つ揃う柱」があれば自分はそこを受けるしかない → 分岐が激減。
4. **着手順序**
   TT の手 → 勝ち手 → 中央寄り → killer / history。良い順序は刈り効率を桁で変える。
5. **対称性圧縮（D4・8 重）**
   盤面を正規化してから TT 照会。実効状態空間が約 1/8。
6. **反復深化 ＋ PVS（Principal Variation Search）**
   時間制御と、前反復の手順による順序改善。
7. **評価関数**：脅威カウント ＋ **奇偶（パリティ）ヒューリスティック**（※確立理論ではなく仮説・方針）
   作業仮説として「その脅威が何段目で完成するか」の偶奇が勝敗に効く、と見て評価に取り込む。
   ただしこれは Connect Four の奇/偶脅威理論（2D・7 列・先手が奇数段という構造に依存）を **そのまま移植できる保証はない**。
   16 柱・3 次元の Score Four では脅威の段の偶奇関係がはるかに複雑で、「立体版の理論」は未確立。
   → **要適応・要検証のヒューリスティック**として扱い、参照実装・自己対戦で効果を測ってから重み付けを決める。
   qweral 氏の戦略 → 評価関数設計の流れと同じく、ここが強さを左右しうると見込むが、確立した本質と断定はしない。

> **労力配分の勘所**：3 の脅威枝刈りが効くと、中盤で局面がそのまま終端（勝ち/負け/引分）まで解けることが多い。
> → 評価関数が効くのは主に序盤〜中盤の入口のみ。労力はまず **2＋3＋4** に振り、評価関数は後回しが正解。

## 5. ビルド順（依存 DAG）

1. **コア**：ビットボード表現＋着手生成＋勝利判定（76 ラインマスク）
   → 総当たり参照実装と突き合わせ、**証明可能なレベル**まで固める。
2. **α-β エンジン**：置換表＋対称性圧縮
   → この時点で **②対戦・解析 AI** が完成。
3. **定石**：2 を根から深く読ませ、最善手順（PV）＋評価値を保存
   → これが **①定石**（自動生成）。
4. **（任意・後で）AlphaZero**：2 が頭打ち、または学習目的のときだけ。

## 6. 言語の判断

- **絶対的な「最強」**を突き詰めるなら、探索速度＝深さ＝強さなので**コンパイル言語（Rust / C++）**が効く。
- ただし「最小労力」とのバランスでは：
  1. まず**移植しやすい Python**でコア＋α-β＋TT＋脅威枝刈りまで作る
  2. 強さと速度を計測
  3. **深さが足りなければホットループだけ Rust / C++ へ移植**
- 補足：この規模なら C++ で最初から書いても手間は大差なく、**C/C++ を多用する今後の仕事の予行演習**を兼ねられる。

---

## 7. 非交渉の原則：コアの正しさ

- **コア（着手生成・勝利判定）のバグは、下流（学習・評価・定石）を静かに全部汚染する。**
- LLM 生成コードは特に勝利判定を微妙に間違える。
- → **契約テストでコアを固めてから探索へ進む「センサー先行」**を徹底する。

## 9. 進捗（2026-06 時点）

この設計に沿って以下を実装済み（詳細は `README.md` / `CLAUDE.md` / `docs/eval_measurements.md`）。

- **コア**：検証済み（ビットボード・76 ライン・増分勝利判定・着手生成）。
- **探索**：4 章の 1〜6 をすべて実装（α-β・置換表・脅威枝刈り・着手順序・D4 対称性圧縮・
  反復深化＋PVS・時間制御）。全幅 negamax 基準の契約テストで値一致を保証。
- **評価（4.7）**：パリティ仮説を**自己対戦で検証**。多シード集計・トーナメントで mode=ALL・
  weight=-8 が頑健に勝ち越すことを確認し既定に採用（depth4 集計 ≈ 0.59）。精緻化案
  （最下段のみ／即着手可能のみ）はベースライン未満で棄却。即時脅威の加点も不採用。
  → 6 章「ホットループだけ移植」の判断に従い **Rust/PyO3 へ移植**（コア＋探索＋評価、
  Python と同値を言語横断契約テストで保証、探索は約 50〜60 倍）。
- **定石生成**：実装済み（5 章の 3 / 4 章の根からの深い反復深化と PV・評価値の保存、
  `book.py` / `data/opening_book.json`）。**Web 対局アプリ**（WASM 化）も実装。
- **詰み探索（Phase 7）**：通常の評価ベース αβ とは別の解析モードとして、零評価（D4 不変）の
  反復深化で強制勝ち/負けの最短手数・PV を読み切る（`solve.py`）。詰み問題の自動生成
  （`problems.py`）も実装。**Rust 移植（`search::solve`）＋ WASM 公開（`sf_solve`）＋ Web の
  「詰み探索」UI** まで通し、Python/Rust 同値を契約テスト・node で確認。別アルゴリズムの
  **df-PN**（`dfpn.py`）も実装・計測したが、詰み証明は高速な一方で詰みなし反証で劣り最短手数/PV
  を返さないためドロップイン置換にせず資産として保持（[`benchmarks/dfpn.md`](benchmarks/dfpn.md)）。
- **強化**（[`docs/roadmap.md`](roadmap.md)）：計測基盤（`analyze`／固定時間ベンチ）、着手順序強化
  （killer/history でノード −37〜50%）、u16 move mask（NPS +12〜15%）を採用。Threat Quiescence・
  Aspiration Window は計測で中立/悪化につき不採用（資産として保持）。
- **強化 Phase 8（軽量学習評価）**: 深い αβ を教師にした D4 不変・整数線形評価を実装・計測。
  教師スコアへの当てはまりは既定パリティより良い（R² 0.78 vs 0.55）が、固定時間自己対戦では
  全条件で負け越し（winrate 0.22〜0.32）＝「fit ≠ strength」。**不採用・資産として保持**
  （[`roadmap.md`](roadmap.md) / [`eval_measurements.md`](eval_measurements.md) 仮説5）。
- AlphaZero は未着手（α-β がまだ頭打ちでないため、設計通り後回し）。学習評価は上記 Phase 8 で
  線形モデルを試し不採用。次の派生候補は勝敗教師のロジスティック学習。
