# CLAUDE.md

Claude Code 用のプロジェクト文脈・規約。実装の前に必ず参照すること。

## このプロジェクトは何か

立体4目並べ（**Score Four / 重力あり 4×4×4**）の **対戦・解析エンジン**と**定石**を作る。
最優先ゴールは **最小労力で最強の解析エンジン**で、手法は **α-β 探索中心**。
方針の全体像は `docs/design.md`、ゲーム仕様は `docs/rules.md` にある。**着手前に両方読むこと。**

## 現状サマリ（2026-06 時点）

コア → 探索一式 → 計測駆動の評価 → Rust 高速化 → **定石生成** まで実装済み。
設計の主要成果物（②対戦・解析 AI と ①定石）が揃った。

- **コア**（`lines.py` / `board.py`）：76 ライン・ビットボード・着手生成・増分勝利判定・
  `winning_moves`。総当たり参照実装との契約テストで固定済み。
- **探索**（`search.py`）：negamax + α-β + 置換表 + 脅威枝刈り + D4対称性圧縮 +
  反復深化 + PVS + 時間制御。全幅 negamax を基準にした契約テストで値一致を保証。
- **評価**（`evaluate.py`）：自己対戦（`selfplay.py`）で計測し、奇偶パリティ
  （`parity_eval`, weight **-8**, mode ALL）を `default_eval` として採用。計測ログは
  `docs/eval_measurements.md`。評価は **D4 対称不変必須**（対称性 TT の前提）。
- **Rust 移植**（`rust/`, PyO3 拡張 `score_four_rs`）：コア＋探索＋評価を移植。Python と
  同値を**言語横断契約テスト**で保証し、探索は **Python 比 約50〜60倍**。評価のパラメータ化
  と高速トーナメント（`play_match`）も実装。
- **定石**（`book.py`, `scripts/generate_book.py`, `data/opening_book.json`）：root から
  `max_plies` 手まで D4 正規化で列挙し、各局面の固定深さ探索の最善手＋評価値を保存。
  **エンジン出力の派生物**。照会 `book_move` / `choose_move`（book になければ探索）。
- テストは全て緑、ruff / clippy / cargo fmt クリーン。次は精度向上（より深い定石・評価の改良）。

## 絶対に外してはいけない前提

1. **対象は重力あり版（Score Four）。Qubic（自由配置）ではない。**
   コマは選んだ柱の最下段に落ちる。任意の高さには置けない。着手＝「柱の選択」のみ。
2. **コアの正しさが全ての土台。**
   着手生成・勝利判定のバグは探索・学習・定石を静かに全部汚染する。
   → **総当たり参照実装との契約テストでコアを固めてから探索へ進む（センサー先行）。**
3. **勝利ラインは全 76 本**（軸 48・平面の斜め 24・立体対角線 4）。
   立体対角線（4 本）と段をまたぐ斜めの実装漏れに注意。

## 実装の順序（この順で進める）

1. ✅ `lines.py`：76 ラインを生成。総当たりで全列挙し、本数 76 をテストで保証。
2. ✅ `board.py`：ビットボード（各色 64bit）、着手生成、増分勝利判定。
3. ✅ `tests/`：ナイーブ参照実装と、ランダム対局で**全局面一致**を property test で検証。
4. ✅ `search.py`：negamax + α-β → 置換表（Zobrist）→ 脅威枝刈り → 着手順序 →
   対称性圧縮（D4・8 重）→ 反復深化 + PVS（＋時間制御）。
5. ✅ `evaluate.py`：脅威カウント ＋ 奇偶パリティ。**自己対戦で計測してから採用**
   （ALL/-8）。確立理論ではなく経験的に検証したヒューリスティック（後述）。
6. ✅ Rust 移植（`rust/`）：コア＋探索＋評価。Python と同値を契約テストで保証（任意・速度用）。
7. ✅ 定石生成（`book.py`）：root から `max_plies` 手まで D4 正規化で列挙し、固定深さ探索の
   最善手＋評価値を保存。`generate_book` / `book_move` / `choose_move`。

## セットアップ（Rust 拡張）

Rust 拡張 `score_four_rs` はビルドして使う。**未ビルドでも純 Python のスイートは緑**
（言語横断テストは `importorskip` で skip）。

```sh
cd rust && maturin build --release \
  && pip install --force-reinstall --no-deps target/wheels/score_four_rs-*.whl
```

テストは `PYTHONPATH=src pytest`（`src` レイアウト）。Rust 側は `cd rust && cargo clippy && cargo fmt`。

## コーディング規約

- Python 3.11+。まず Python で正しく作り、速度が律速なら**ホットループのみ** Rust へ移植する（早すぎる最適化はしない）。Rust は PyO3 0.29 + maturin。
- 盤面表現・セルのインデックス規約は `board.py` の冒頭 docstring に**唯一の定義**として書き、Python・Rust 双方がそれを参照する（`index = z*16 + y*4 + x`）。
- 公開関数には型注釈と docstring。副作用のある関数と純粋関数を混ぜない。
- 乱数・探索の非決定性はテストで排除（シード固定、同一局面で同一手）。
- **評価関数は D4 対称不変であること**（z=高さは D4 で不変なので、ライン占有・高さ・
  パリティで作れる）。非対称な評価は対称性圧縮 TT と矛盾する → `test_symmetry.py` で検証。
- Rust と Python で**同じアルゴリズム・同じ着手順序・同じ TT 意味論・同じ PVS**を保ち、
  契約テストで結果一致を担保する。

## テストの方針（最重要）

- **新しいコア機能は、参照実装との一致テストを書いてからマージする。**
- 既知局面（即詰み・ダブルリーチ・引分確定）の回帰テストを増やしていく。
- 探索は **全幅 negamax を基準**に、α-β＋各最適化が**同値**を返すことを契約テストで保証。
- Rust 実装は **言語横断契約テスト**（`test_rust_core.py` / `test_rust_search.py`）で
  Python 参照と全局面・全評価層の一致を保証する。
- 評価ヒューリスティックは**自己対戦で測ってから採用**。単一シードを鵜呑みにせず
  **多シード集計**で頑健性を確認する（`docs/eval_measurements.md` の経緯参照）。
- `pytest` を緑にしてから次のレイヤへ進む。

## やらないこと

- 最初から AlphaZero に手を出さない（α-β が頭打ちになるまで不要。現状まだ頭打ちではない）。
- 評価ヒューリスティックを**計測せずに**既定へ採用しない（もっともらしい改変が中立〜悪化する例あり）。
- コアの契約テストが緑になる前に上位レイヤを書き始めない。
- Rust と Python の探索結果がずれた状態でマージしない。
